Quick summary:
Command	                Purpose
kubectl create -f	Creates a Deployment from a YAML file
kubectl get deployments	Lists all Deployments and their status
kubectl get replicaset	Shows ReplicaSets created by the Deployment
kubectl get pods	Displays Pods managed by the ReplicaSet

Option to see all objects instances: kubectl get all

Quick mental model
Level	               Object	           Purpose
Deployment	Manages ReplicaSets	Handles updates and rollbacks
ReplicaSet	Manages Pods	        Ensures correct replica count
Pod	        Runs containers	        Each Pod = 1 nginx container here

Deployment strategy:


Recreate - when kill all existing containers and then deploy new version . So downtime is existing until new containers are beeing up.

Rolling Update - when we destroy first container and instead start new container , then we destroy second container andup new version of second container. Default deployment strategy.



We can modify our deployment.yaml with new version of IMAGE in the file and run "apply":

> kubectl apply -f deployment.yaml

Step-by-step behavior: 

Kubernetes detects a change
It compares the new YAML with the existing Deployment.
If the Pod template (like the container image) changed, Kubernetes knows it must update the Pods.


A new ReplicaSet is created
Your Deployment automatically creates a new ReplicaSet with the updated image.
The old ReplicaSet still exists — it’s kept temporarily for rollback.

Rolling update begins
Kubernetes starts replacing Pods gradually:

Spins up new Pods with the new image.

Waits for them to become Ready.

Terminates old Pods one by one.

This ensures zero downtime if your readiness probes are configured correctly.

You can watch it live:

kubectl rollout status deployment/<deployment-name>

Old ReplicaSet remains for rollback
After the rollout finishes, the old ReplicaSet stays in the background.
You can see it with:

kubectl get replicaset

And rollback if needed:

kubectl rollout undo deployment/<deployment-name>

Kubernetes records the change:

kubectl rollout history deployment/<deployment-name>


Action	Result
kubectl apply -f deployment.yaml	Updates Deployment spec
Pod template changed (e.g., new image)	Creates new ReplicaSet
Rolling update	Gradually replaces old Pods
Old ReplicaSet	Retained for rollback
Rollout history	Revision number increases


How long does the old ReplicaSet stay alive?
Forever — until YOU delete it.

Kubernetes never automatically deletes old ReplicaSets created by a Deployment.
They remain in the cluster as long as the Deployment exists, because they are needed for:

rollbacks

rollout history

tracking previous revisions

BUT if we run like this: kubectl set image deployment/myapp-deployment nginx=nginx:1.9.1

It will do the same BUT the yaml file will still have the old image version !!!!

Immediate effect
Kubernetes updates the Deployment object directly in the cluster.

It modifies the Pod template inside the Deployment to use nginx:1.9.1.

A new ReplicaSet is created, and a rolling update begins automatically.

The change is stored in the cluster, not in your local YAML file.


When we do Rolling Update - cause previous replica set still be up (until we delete it) if needed we can do ROLL BACK:

kubectl rollout undo deployment/myapp-deployment


The truth: Rollback uses the previous ReplicaSet, not your YAML
A Deployment keeps a history of ReplicaSets, each representing a version of your Pod template.

When you run rollback:

Kubernetes finds the previous ReplicaSet (the one before your last update)

Scales it up to the desired number of replicas

Scales the current ReplicaSet down to 0

Marks the rollback as a new revision in history

Your YAML file is not involved in this process.

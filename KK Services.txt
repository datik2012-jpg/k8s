/////////////////////////Services in k8s ///////////

Node -> Service -> Pod

NodePort - when we want our app in the Pod be accessible outside the node , so we need configure Service which can map port on the Pod ,then we can curl with NOde ip + this port and service will send the trafict to the Pod.
Services is using LABELS of the Pod.In case we have in the NOde multiple Pods with the same label like "myapp".Uses Algorith:Random and SessionAffinity:Yes.No need to do something.

NodePort is only required when traffic comes from outside the cluster.

NodePort - in range 30000-32767

You want to open your app in a browser on your laptop

You want to curl the service from outside Kubernetes

You want to expose a web app to the public

////////////////////////////K8s Service  ClusterIP ////////////////


A ClusterIP Service is the default and most common Service type in Kubernetes. It provides a stable, internal-only virtual IP that other Pods inside the cluster can use to reach your application. It is the foundation of Pod‑to‑Pod communication and internal load balancing.
What ClusterIP actually is (core definition)
A ClusterIP Service exposes your application on a virtual IP address that is reachable only inside the cluster.
Traffic sent to this IP is load‑balanced across all Pods that match the Service’s selector.

Your intuition is exactly right, Dani — the worker Pod does NOT need any ports exposed, because it is not a server. It only initiates connections (to Redis, DB, API, etc.), so it behaves like a client inside the cluster.

And here’s the key:

In Kubernetes, Pods talk to other Pods using Service DNS names — not ports on nodes, not Pod IPs.

So your worker Pod will talk to the DB using the DB Service name, not the DB Pod IP.

Your worker simply connects to:
db-svc:5432
Kubernetes DNS resolves that name to the current DB Pod, no matter which node it runs on.

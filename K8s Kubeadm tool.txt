What is kubeadm?
kubeadm is a Kubernetes tool that helps you bootstrap a production‑grade Kubernetes cluster.
It doesn’t install Kubernetes for you — it initializes and configures the control plane and worker nodes.

Think of kubeadm as the installer and cluster builder for Kubernetes.

Why kubeadm is needed
kubeadm is used because it:

Creates the control plane (API server, scheduler, controller manager)

Generates certificates for secure cluster communication

Bootstraps worker nodes and joins them to the cluster

Configures networking (CNI)

Ensures Kubernetes best practices for cluster layout and security

It’s the official and recommended way to create Kubernetes clusters on bare metal or VMs.

Cloud providers (EKS, AKS, GKE) hide kubeadm from you — but under the hood, they use similar logic.



Multipass is a lightweight virtual machine manager created by Canonical (the makers of Ubuntu).
It’s designed to give you instant Ubuntu VMs with a single command — much simpler than Vagrant or VirtualBox.

What Multipass is
Multipass is a mini‑cloud on your laptop.
It lets you launch Ubuntu VMs extremely fast, using:

VirtualBox

Hyper‑V

QEMU

macOS Hypervisor

It’s like having a tiny AWS EC2 on your machine.

🚀 Step 1 — Launch 3 VMs with Multipass
Run these commands on your host machine:

bash
multipass launch --name master --mem 4G --disk 20G
multipass launch --name worker1 --mem 4G --disk 20G
multipass launch --name worker2 --mem 4G --disk 20G
Check they’re running:

bash
multipass list
🛠 Step 2 — Install MicroK8s on all nodes
Master:
bash
multipass shell master
sudo snap install microk8s --classic
sudo usermod -aG microk8s ubuntu
sudo microk8s status --wait-ready
Workers:
bash
multipass shell worker1
sudo snap install microk8s --classic
sudo usermod -aG microk8s ubuntu
sudo microk8s status --wait-ready
Repeat for worker2.

🔗 Step 3 — Get the join command from the master
Inside master:

bash
microk8s add-node
You’ll get output like:

Code
microk8s join 192.168.64.2:25000/abcdef1234567890
Copy that command.

🤝 Step 4 — Join worker nodes to the cluster
On worker1:

bash
sudo microk8s join 192.168.64.2:25000/abcdef1234567890
On worker2:

bash
sudo microk8s join 192.168.64.2:25000/abcdef1234567890
🌐 Step 5 — Enable networking & dashboard
On the master:

bash
sudo microk8s enable dns storage dashboard
Optional but recommended:

bash
sudo microk8s enable ingress
📦 Step 6 — Use kubectl
MicroK8s includes its own kubectl:

bash
microk8s kubectl get nodes
You should see:

Code
master     Ready
worker1    Ready
worker2    Ready

What kubeadm does in production
Each item starts with a Guided Link so you can explore it deeper.

- Bootstrap control plane
Creates API server, scheduler, controller manager.

- Join worker nodes
Adds nodes to the cluster securely.

- Generate certificates
Handles TLS for cluster components.

- Upgrade clusters
Provides safe upgrade commands.

- Install CNI plugins
Calico, Flannel, Cilium, etc.

What kubeadm does NOT do (you must handle)
Load balancer for control plane

etcd backup/restore automation

Monitoring (Prometheus/Grafana)

Logging (ELK, Loki)

Ingress (NGINX, Traefik)

Storage (Ceph, Longhorn, NFS)

Node lifecycle automation

This is why kubeadm is powerful but requires expertise.

kubeadm gives you full control, but also full responsibility.


When kubeadm is a good production choice
Use kubeadm in production when you want:

On‑premise Kubernetes

Bare‑metal clusters

Edge clusters

Full customization

No cloud vendor lock‑in

High performance networking (Calico, Cilium)

To learn real cluster internals

This is why many DevOps engineers prefer kubeadm for labs and real deployments.

When kubeadm is not ideal
Avoid kubeadm if you want:

Automatic upgrades

Built‑in HA

Managed control plane

Easy scaling

Minimal maintenance

In that case, use:

AKS

EKS

GKE

OpenShift

K3s (for edge)

MicroK8s (for simplicity)
